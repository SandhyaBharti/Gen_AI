# monolithic and microlithic architecture  (part 1)
# monolithic architecture -> everything is tightly coupled depended on each other 
issue 1: single code base-> we will build + run + test + deploy together
disadvantage:- single point failure-> if one file will crash then other will also get down 
issue 2:- deployment bottleneck -> i have to add credit card feature in the payment gateway then after adding it again i have to  re deploy the whole project (issue ko deployemnt bottleneck issue bolte hain)
issue 3 -> scaling :- i have multiple pages lots of users uses at the same time, so that i have to scale up the pages (each and every page) bcz it's a small website

# Microservice architecture
every module is losely coupled, they are not too much depended on each other
we can build them individually + test + Deploy + 

@monolithic
s1[individually] -> s2[individually] -> s3[individually] 
all are not dependent on each other 

*shared database-> each collection under one database (table can be different but single database)

@microlithic
s1->s2->s3
every service can have their own database

# we use design pattern 
The name is "Strangler Design Pattern"
IN SDP we gradually replace a legacy system with new one -> without shuting everything down at once. 

request -> SDP:-
=>90%->old arch (shift 90% arch to old one)
=>10%->new arch (shift 10% arch to new one)

#Transaction: SAGA : the design pattern hanles transactions across multiple microservices without using single global DB transaction
Core idea: instead of core big transaction, u break it into small local transaction If something fails:-> u undo previous step(compensation)
2 types of SAGA:=> 
(i) Choreography(event-based):- services talk via events (no central controller),
                                    flow->order created->payment listens->payment success->inventory listens
(ii) Orchestration(central control):- one service controls the flow(saga orchestration)
                                        flow-> orchestratior->call order->then payment->then inventory

# Data Consistancy (Outbox) :=> ex: 
when a user place order, u need to do 2 things:- (i) save the order in database (ii) send the event -> "order created"(so that payment service can act)
Problem(without outbox): what if:- 
*order saved
*event sending fails
-> now ur system is broken(order is about no payment )

with outbox pattern:-
trusted of sending the event directly 
step 1: save both in one db transaction, 
        -> order -> order table
        -> event -> outbox table

    "event": "order created",
    "orderID": "123"
